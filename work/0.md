# Telltale Host Ownership Contract

## Goal

Strengthen Telltale's host-runtime contract so fragment and session ownership are explicit, transferable, auditable, and testable.

This design imports the useful ownership ideas from Aura into Telltale's embedding surface without turning Telltale into an Aura-style application framework.

The main change is not to Telltale's session-type theory. The main change is to the Rust host boundary around VM execution, reconfiguration, and effect integration.

Additional guidance borrowed from Aura's choreography/runtime integration:

- production ownership should be aligned with composition metadata rather than ad hoc host structure
- generated role runners may remain useful for tests and tooling, but should not be treated as the production ownership boundary
- reconfiguration should flow through one sanctioned manager path
- delegation should produce auditable transfer artifacts or receipts
- stale-owner rejection should be generation-based, not just owner-label-based
- boundary-scoped ownership should be able to attenuate authority from full-session to fragment-scoped access
- raw VM-open helpers should stay runtime-internal; higher layers should use an owned-session wrapper

## Non-Goals

- Do not import Aura's full repo-wide ownership taxonomy as a required framework-wide doctrine.
- Do not add Aura-specific workflow publication wrappers, UI handoff rules, or application macros to Telltale core.
- Do not replace Telltale's existing admission, determinism, or parity model.
- Do not weaken the current split where Telltale owns protocol typing and the host owns materialization.

## Current Baseline

Telltale already has several pieces this work should build on:

- Delegation and reconfiguration harmony are already theory-level concepts.
- `DelegationWF` already states protocol-side safety conditions for delegation.
- The VM already has capability/admission gates.
- The VM already performs some ownership checks, such as endpoint ownership on close.
- The host bridge already distinguishes VM-owned semantics from host-owned effects.

What is missing is a stronger and more explicit host-runtime contract for:

- who currently owns a fragment or session,
- how ownership transfer occurs,
- how stale owners are rejected,
- how async host events are routed before state mutation,
- what happens when an owner dies or abandons responsibility.

## Design Summary

Telltale should introduce a first-class host ownership contract with these rules:

1. Each active fragment has exactly one current local owner.
2. Ownership transfer is explicit and uses a transfer object or receipt, not ad hoc field mutation.
3. Ownership and capability are distinct.
4. Host events must enter through canonical ingress and be routed to the current owner before session mutation.
5. Stale-owner actions fail closed with typed diagnostics.
6. Owner loss or failed transfer must produce explicit terminal session behavior.
7. Reconfiguration APIs must enforce ownership transfer, capability scope checks, and coherence checks together.
8. These properties must be observable in docs, runtime APIs, assertions, and simulator tests.

## Core Model

### Ownership Unit

The ownership unit should be a fragment-scoped VM runtime unit.

Default interpretation:

- a choreography without link composition metadata is one fragment
- a linked composition may produce multiple ownership units
- delegation may move ownership of an endpoint or fragment-scoped sub-unit, but the API must make the move explicit

This aligns host ownership with Telltale's existing composition and reconfiguration model.

Where composition metadata exists, the runtime should prefer it as the canonical source for fragment boundaries and ownership-unit derivation instead of reconstructing boundaries from ad hoc host logic.

### Ownership Invariant

For every active fragment:

- there is at most one current owner
- only the current owner may drive owner-routed effects or mutate session-local host state
- any prior owner handle becomes stale after transfer
- no API path may create overlapping owners

Prior-owner invalidation should be generation-based. Reusing the same textual owner label must not preserve authority if the ownership generation or boundary scope changed.

### Ownership vs Capability

These must be separate concepts:

- ownership answers who currently controls the fragment
- capability answers what work that owner is allowed to perform

Admission and runtime capability checks remain necessary, but they do not substitute for ownership.

The runtime should represent current ownership as a typed capability or token carrying:

- session / fragment identity
- owner label
- ownership generation
- authorized scope

### Canonical Ingress

External events must not mutate session state directly.

They must first become ingress events, for example:

- transport envelopes
- timer events
- host commands
- delegation / reconfiguration deliveries

The runtime must route these through the current owner before state mutation occurs.

Ingress checks should validate both:

- task/session binding, so the caller is acting in the correct runtime context
- boundary authorization, so fragment-scoped owners cannot perform full-session operations

### Terminality

Owner loss must not leave responsibility ambiguous.

If the current owner dies, abandons transfer, or violates the ownership contract, the runtime must produce an explicit typed outcome, typically by faulting or cancelling the affected fragment/session according to the runtime policy.

Terminal state requirements:

- no silent abandonment
- no parallel terminal publishers
- no non-terminal stale owner after transfer

## Borrowed Implementation Patterns

These patterns are worth borrowing because Aura implements them concretely rather than only describing them.

### Generation-Based Owner Capabilities

Aura does not rely only on owner labels. It increments a generation on each ownership transfer and rejects stale capabilities even when the owner label stays the same but scope changes.

Telltale should strongly consider the same shape.

### Owned Session Wrapper

Aura keeps the raw manifest/session-open path runtime-internal and exposes an owned-session wrapper that:

- starts a runtime-owned session
- claims owner capability
- derives the routing boundary from manifest metadata
- checks ownership/boundary constraints before each ingress operation

Telltale should prefer a similar wrapper pattern for production integration guidance.

### Boundary-Scoped Ownership

Aura distinguishes:

- full-session ownership
- fragment/boundary-scoped ownership

After delegation, the new owner may intentionally lose full-session authority and retain only the delegated boundary scope. That is a useful pattern for Telltale reconfiguration APIs.

### Coupled Transfer of Session Owner and Fragment Registry

Aura transfers the runtime session owner and fragment ownership together. Telltale should preserve this idea: do not move the logical owner while leaving fragment records behind, and do not move fragment records without updating the current owner capability.

### Rollback on Live Handoff Failure

Aura's live delegation path transfers the active owner first, then performs reconfiguration, and rolls owner/boundary state back if the reconfiguration step fails.

Telltale should adopt an explicit rollback story for live ownership handoff.

Rollback should be claim-specific. If a runtime stages ownership records or fragment claims during open/transfer and later fails, it should release only the records created by that attempt rather than tearing down every fragment bound to the session wholesale.

### Audit Fact / Receipt Persistence

Aura persists a delegation fact and also returns a typed receipt/witness. Telltale should at least support typed receipts, and should consider trace/audit artifacts for reconfiguration events when host runtimes need auditability.

## Proposed Rust Surface Changes

This section describes the intended API direction. Exact names may change during implementation.

### Ownership Types

Add explicit ownership-related runtime types in `telltale-vm` or the composition/runtime layer:

- `FragmentOwnerId`
- `OwnershipEpoch`
- `OwnershipCapability`
- `OwnershipReceipt`
- `OwnershipTransfer`
- `OwnershipError`
- `FragmentIngress`

`OwnershipTransfer` or `OwnershipReceipt` should be consumed to complete transfer. The old owner handle should no longer be valid afterward.

### Session / Fragment Store

Extend runtime session or composition state to track:

- current owner id
- ownership epoch / generation
- current authorized scope
- ownership status
- optional transfer-in-progress state
- terminal reason if ownership failure caused fault or cancellation
- enough metadata to roll back only the staged claims created by one failed operation

### Reconfiguration APIs

Strengthen link/delegate/reconfiguration surfaces so they require:

- proof of current ownership
- explicit transfer payload / receipt
- capability scope input if capability is part of the move
- coherence or `DelegationWF`-aligned validation before commit

Forbidden pattern:

- direct owner-field rewrites in runtime code

The intended architecture is a sanctioned manager-style path for reconfiguration rather than many distributed ownership mutation call sites.

### Host Callback Contract

Clarify and, where needed, tighten `EffectHandler` / VM bridge rules:

- callbacks remain synchronous
- async transport / storage / journal work remains outside callback paths
- host callbacks may only operate on state scoped to the current owned fragment
- callback-originated mutation must be rejectable if the caller is stale or not the current owner

The preferred public integration shape is:

- raw/session-open helpers remain runtime-internal or low-level
- production-facing helpers return an owned session / owned fragment wrapper that enforces ingress checks

### Host Assertions

Extend `VMConfig.host_contract_assertions` or adjacent hardening controls to check:

- one owner per active fragment
- no owner overlap during transfer
- ingress is routed before mutation
- stale-owner actions are rejected
- handler identity / owner identity stability across commit boundaries where required

## Testing Strategy

These stronger properties need dedicated tests, not just doc updates.

### Unit Tests

Add Rust unit tests for:

- owner claim succeeds for an unowned fragment
- second concurrent claim is rejected
- transfer invalidates the prior owner
- transfer invalidates the prior owner even when the owner label is unchanged but generation or scope changed
- stale-owner send / receive / invoke / release paths fail closed
- ingress events cannot mutate state without owner routing
- owner death transitions the session to the correct terminal status

### Integration Tests

Add runtime integration tests for:

- reconfiguration / delegation with explicit ownership transfer
- failed transfer rollback or fail-closed behavior
- capability accepted but ownership rejected
- ownership accepted but capability rejected
- no overlapping owners during link/delegate flows
- manifest/composition-derived fragment boundaries match runtime ownership-unit boundaries
- generated runner or helper paths cannot bypass the sanctioned production ownership path
- fragment-scoped delegated owners retain delegated-boundary access but lose full-session ingress
- failed open/claim rollback releases only the claims created by that open attempt

### Simulator Tests

Add simulator scenarios for:

- delayed transfer receipt arrival
- duplicated transfer attempts
- stale-owner replay after successful transfer
- owner crash during transfer
- concurrent network/timer events during ownership handoff
- transfer receipt auditing and observability under reconfiguration

### Parity / Conformance Tests

Ensure stronger ownership rules do not silently drift from existing parity assumptions.

Add or update tests for:

- cooperative path behavior under ownership assertions
- threaded / advanced path fallback or rejection if ownership constraints are violated
- canonical replay artifacts reflecting ownership-related fault outcomes
- determinism of ownership failure diagnostics where they are part of observable behavior

## Failure Model

Ownership failures should be typed and machine-visible.

Suggested error categories:

- `NotCurrentOwner`
- `OwnershipEpochMismatch`
- `TransferAlreadyInProgress`
- `OverlappingOwnerClaim`
- `IngressBypassed`
- `CapabilityScopeMismatch`
- `OwnerAbandoned`
- `TransferValidationFailed`

These can map into existing VM/runtime error surfaces if that is cleaner than exposing a fully separate public enum immediately.

## Phase 1 Decisions

Phase 1 resolves the design-shape questions needed before implementation.

### Normative Ownership Model

Telltale will use a layered ownership model:

- session ownership is the authoritative runtime lifecycle owner model
- fragment ownership is the routing and reconfiguration scope model inside a session
- every active session has exactly one current owner capability
- that owner capability may be full-session scoped or attenuated to one or more fragment scopes
- fragment-scoped access does not imply full-session ingress rights

### Ownership Unit Decision

The ownership unit is layered rather than fragment-only or session-only.

- the session is the unit of owner identity, generation, and lifecycle responsibility
- fragments are the unit of routed authority and reconfiguration boundary
- ownership transfer always updates the current session owner capability
- if fragment ownership exists for the session, it must move together with the owner capability

### Minimum Runtime Data Model

Phase 1 chooses the following minimum data model:

- `session_id`
- `owner_label`
- `ownership_generation`
- `ownership_scope`
  - `Session`
  - `Fragments(BTreeSet<String>)`
- `fragment_ids`
- `transfer_state`
  - `Idle`
  - `TransferInProgress`
- `ownership_terminal_reason`
- `staged_claims` for claim-specific rollback

### Ownership vs Capability at API Boundaries

Phase 1 chooses this split:

- admission/capability gates answer whether a runtime profile or operation class is allowed in principle
- ownership capability answers whether this caller is the current live owner right now
- boundary scope answers which fragment/session surface the current owner may drive

Host-facing mutation or ingress APIs should require current ownership capability, not just a protocol id or admission result.

### Ownership Error Taxonomy Decision

Phase 1 adopts this ownership-oriented taxonomy as the target public/runtime language:

- `NotCurrentOwner`
- `OwnershipEpochMismatch`
- `CapabilityScopeMismatch`
- `OverlappingOwnerClaim`
- `TransferAlreadyInProgress`
- `IngressBindingMismatch`
- `OwnerAbandoned`
- `TransferValidationFailed`
- `TransferRollbackFailed`

Mapping decision:

- ownership and boundary violations should surface first as typed runtime/session-ingress errors
- where they cross the VM host boundary, they may map into existing contract/determinism/topology failure buckets rather than forcing a brand-new VM public error surface immediately
- close/open/reconfiguration failures should remain machine-visible and fail closed

### Composition Metadata Decision

Composition metadata is the canonical source of ownership-unit derivation and fragment keys whenever it is available.

Default derivation:

- no link metadata -> one fragment keyed by protocol id
- link metadata present -> one fragment per bundle id

Hosts should not reconstruct fragment boundaries independently when manifest metadata already defines them.

### Ownership Capability Object Decision

Telltale should represent live ownership as a generation-bearing capability/token object.

Minimum fields:

- `session_id`
- `owner_label`
- `generation`
- `scope`
- optional `fragment_keys`

Reusing the same owner label after transfer does not preserve authority because generation and scope are part of validity.

### Public Integration Shape Decision

Telltale should expose a preferred owned-session / owned-fragment wrapper for production host integrations above lower-level raw session-open helpers.

The intended split is:

- low-level helpers may remain available for tests, scaffolding, or narrow integrations
- production guidance and safer defaults should flow through owned wrappers that enforce ingress and transfer checks

## Remaining Open Questions

- Is endpoint-level transfer represented as a special case of fragment transfer, or as a distinct ownership object?
- Which ownership failures should map to `Faulted` versus `Cancelled`?
- Should ownership epoch be observable in traces, or only in debug / assertion modes?
- Which ownership diagnostics belong in stable public API versus internal hardening lanes?

## Acceptance Criteria

This work is complete when all of the following are true:

- Telltale has a documented host ownership contract for fragments/sessions.
- Ownership transfer is explicit in runtime APIs.
- Admission and ownership are clearly separated in docs and code.
- Stale-owner operations fail closed with typed diagnostics.
- Owner loss produces explicit terminal behavior.
- Reconfiguration paths use sanctioned transfer APIs rather than ad hoc mutation.
- Docs and tests cover the new contract thoroughly.

## Recommended Order

1. Finalize the ownership model and API vocabulary.
2. Implement core ownership state and transfer enforcement.
3. Wire reconfiguration/delegation through the sanctioned transfer path.
4. Add assertions and tests.
5. Update docs after code shape stabilizes, then reconcile wording with the final implementation.
